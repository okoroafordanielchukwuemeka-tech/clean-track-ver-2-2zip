PHASE 7.10 — SERVICE CATALOG & IMAGE MANAGEMENT

IMPORTANT

Do NOT begin Phase 8.

Do NOT modify billing, subscriptions, documents, workflows, or permissions.

This phase focuses ONLY on the Service Catalog experience.

Before implementing anything:

Audit the current Service Management module.

Determine:

1. How services are currently stored.
2. Whether categories already exist.
3. Whether images already exist.
4. Whether branch-specific services are supported.
5. Whether search already exists.
6. Whether service ordering exists.
7. Whether service analytics exist.

If any feature already exists:

Verify it works correctly.

Do NOT rebuild it.

Only fix defects.

---

SECTION 1 — Image Architecture

Do NOT store image binaries inside PostgreSQL.

Implement proper object storage.

Current recommendation:

• Replit Object Storage (current)
OR

Create a storage abstraction so Cloudflare R2 or Amazon S3 can be used later without changing business logic.

Database should store ONLY:

imageUrl

Never store image blobs.

Verify:

Uploading

Replacing

Deleting

Retrieving

Thumbnail generation

Unused image cleanup

---

SECTION 2 — Default Image Library

Examples:

• Shirt
• Trouser
• Suit
• Native Wear
• Gown
• Wedding Dress
• Curtain
• Blanket
• Duvet
• Pillow
• Towel
• Rug
• Carpet
• Sneakers
• Leather Shoes
• Canvas Shoes
• School Uniform
• Children's Clothes

When a user creates a service:

If the service name closely matches a built-in service:

Suggest the default image automatically.

Example:

User types:

White Shirt Premium

Automatically suggest:

Shirt image

Owner can:

Use default image

OR

Upload a custom image

SECTION 3 — Custom Uploads

• JPG
• PNG
• WEBP

Compress automatically.

Generate thumbnail automatically.

Reject oversized files.

Verify mobile uploads.
